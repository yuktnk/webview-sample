# ADR-004: グローバル状態管理ライブラリを使わない

|            |            |
| ---------- | ---------- |
| **Status** | Accepted   |
| **Date**   | 2026-04-01 |

---

## Context

旧実装では Redux + Redux Saga で 18 モジュールが管理されていた。
フルリプレイスにあたり、Zustand・Jotai などのグローバル状態管理ライブラリの採用を検討した。

### 旧 Redux モジュールの分類

| カテゴリ                                        | 該当モジュール                                    | 割合   |
| ----------------------------------------------- | ------------------------------------------------- | ------ |
| A. サーバーデータ（API レスポンスのキャッシュ） | WeeklyReport / MonthlyLookback など 16 モジュール | 約 80% |
| B. 認証・設定（全ページ共通）                   | Config（userId / stores / batchInfo など）        | 約 15% |
| C. ローカル UI 状態（本来の状態管理）           | ItemAnalysis.ui（selectedType）など               | 約 5%  |
| D. トラッキング専用（Reducer なし）             | UI モジュール                                     | 0%     |

**A・B の大半は「状態管理」ではなく「API レスポンスの置き場」になっていた。**
Redux + Saga は API 1 本のために Action / Saga / Reducer の 3 ファイルを必要とし、ボイラープレートが膨大だった。

**C の「本当の UI 状態」は全ページ内で完結しており、グローバルに共有する必要がない。**

### このアプリのアーキテクチャ上の制約

**1 ページ = 1 WebView** であるため：

- ページ間で状態を共有する必要がない
- グローバル Store のメリットが活きる場面がない
- コンポーネントツリーの頂点から `useState` + Context で十分

### Zustand / Jotai を検討した結果

|                            | Zustand                                          | Jotai                                  |
| -------------------------- | ------------------------------------------------ | -------------------------------------- |
| 思想                       | 小さい Store                                     | 原子（atom）単位                       |
| このアプリへの適合         | △ Store の恩恵なし                               | △ atom の恩恵なし                      |
| 導入する価値が生まれる条件 | 複数コンポーネントで深く共有する状態が増えた場合 | 複数ページで同じ UI 状態を使い回す場合 |

現時点でどちらも「使う理由がある」場面が存在しない。
「使ってみたい」以外の理由がなく、シンプルさを損なうだけと判断した。

---

## Decision

**グローバル状態管理ライブラリ（Redux・Zustand・Jotai 等）を使わない。**

旧 Redux 18 モジュールは以下に置き換える：

| カテゴリ            | 旧               | 新                                        |
| ------------------- | ---------------- | ----------------------------------------- |
| A. サーバーデータ   | Redux + Saga     | TanStack Query                            |
| B. 認証・設定       | Redux config     | TanStack Query（`rootRoute.loader` 経由） |
| C. ローカル UI 状態 | Redux            | `useState` / `useReducer` / Context       |
| D. トラッキング     | Redux Middleware | NativeBridge を直接呼ぶ                   |

---

## Consequences

**メリット**

- ボイラープレートがなくなり、API 1 本の追加が query / type / handler / mock の 4 ファイルで完結する
- ページ間の状態共有が構造的に存在しないため、状態の流れが追いやすい
- 新規参画者が Redux の概念を習得しなくても開発できる

**デメリット・制約**

- 将来、複数コンテナ間で深く共有する UI 状態が生まれた場合は、その時点で改めて Zustand / Jotai の導入を検討する
- 旧実装が Redux Middleware 経由で NativeBridge を呼んでいた迂回路を廃止するため、直接呼び出しへの書き直しが必要になる（詳細は [native-bridge.md](../../.claude/rules/native-bridge.md) 参照）
