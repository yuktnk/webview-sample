# ADR-001: フルリプレイスを採用する

|            |            |
| ---------- | ---------- |
| **Status** | Accepted   |
| **Date**   | 2026-04-01 |

---

## Context

旧WebViewアプリは iOS/Android ネイティブアプリ内に埋め込まれた WebView アプリである。
調査時点（2026-04-01）で以下の技術的負債が重なっており、段階的アップグレードの実現可能性を検討した。

### 主な技術的負債

| カテゴリ       | 問題                                                            | 深刻度    |
| -------------- | --------------------------------------------------------------- | --------- |
| 依存関係       | Next.js 9（2019年製）+ React 18 という非公式な組み合わせ        | 🔴 致命的 |
| 依存関係       | formik 0.10 / yup 0.24（現行 API と別物）                       | 🔴 高     |
| TypeScript     | `strict: false` + `any` が 83 ファイル・841 エラーを隠蔽        | 🔴 高     |
| テスト         | ユニットテスト 4 ファイル（395 ファイル中）・E2E テストなし     | 🔴 高     |
| アーキテクチャ | クラスコンポーネント 57 ファイル・Redux `connect()` 46 ファイル | 🟡 中     |
| ライブラリ     | moment.js（公式メンテナンス終了宣言済）が 20 ファイルで使用     | 🟡 中     |

### 段階的アップグレードを選ばなかった理由

段階的アップグレードが通常より困難である構造的な問題が 4 つ重なっている。

**① テストがないためデグレを検出できない**
テストカバレッジが事実上 0% のため、各アップグレードステップでの退行は手動確認にしか依存できない。
20 画面を毎回手動確認することは現実的ではなく、見落としが発生する。

**② 型が緩いためコンパイラが安全網として機能しない**
`strict: false` + `any` 146 箇所の状態では、ライブラリ更新時の影響箇所をコンパイラが教えてくれない。
ランタイムで初めてクラッシュする箇所が大量に発生する。

**③ 変更が絡み合っており独立して進められない**
Next.js を上げると styled-components が壊れ、axios を上げると Saga の書き方が変わる、という連鎖が続く。
全ステップを完了した時点でほぼ全ファイルが書き直されており、フルリプレイスと作業量が変わらない。
それでいて「半分古い・半分新しい」という最も保守しにくい状態が長期間続く。

**④ 段階的アップグレードでは SSR コストが残り続ける**
現状は `getInitialProps` で SSR を試みているが、実際にはサーバー側で Redux ストアにデータを入れておらず、
クライアントの `componentDidMount` で API コールが走る。SSR の恩恵を受けていないにもかかわらず
Node.js サーバーの常時起動コストだけを払い続けている。
段階的アップグレードでは Next.js を維持するため、このコストが解消されない。

### 移行方式

Native 側の変更なしに段階的切り替えが可能。

```
GCP プロジェクト
├── dispatch.yaml        ← URL ごとに旧・新アプリの振り先を制御
├── service: webview     ← 旧アプリ（移行中も触らない）
└── service: webview-v2  ← 新アプリ（1画面ずつ追加）
```

移行済みの画面から順に `dispatch.yaml` を更新するため、移行中も既存機能は壊れない。

---

## Decision

**段階的アップグレードではなく、フルリプレイスを採用する。**

Vite + React 19 による新アプリを別サービスとして構築し、`dispatch.yaml` で画面単位に切り替える（なぜ Vite を選んだかは [ADR-009](./009-vite-react-over-nextjs.md)）。
旧アプリは移行期間中も一切変更しない。

---

## Consequences

**メリット**

- テスト・型安全・アーキテクチャをゼロから正しく構築できる
- SSR をやめることで Node.js サーバーが不要になり、App Engine の静的配信のみになる（インフラコスト削減）
- `dispatch.yaml` による段階切替で、常にどちらかのアプリが完全な状態で稼働する
- 旧アプリを触らないため、移行中に既存機能が壊れるリスクがない

**デメリット・制約**

- 移行期間中は旧・新アプリの 2 系統を並行維持する必要がある
- 旧アプリのドメインロジック（API URL・パラメータ・レスポンス型）をコードから読み取って移植する作業が必要
- 「現在のコードへの慣れ」はリセットされる（これは技術的負債の解消と引き換えに手放すもの）
