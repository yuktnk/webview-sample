# ADR-011: バリデーションを Zod に統一する

|            |            |
| ---------- | ---------- |
| **Status** | Accepted   |
| **Date**   | 2026-04-01 |

---

## Context

旧実装では yup（formik と組み合わせ）と型なし axios が混在しており、バリデーションが一本化されていなかった。

```ts
// 旧実装①：API レスポンスに型がない
axios.get('/core/v2/user/user_info/').then((res) => res.data.result) // res.data は any

// 旧実装②：フォームバリデーションは yup
const schema = yup.object({ storeCode: yup.string().required() })
```

この状態では：

- API レスポンスの型は手書き `interface` で定義するが、実際のレスポンスとの乖離を検知できない
- ルーティングのクエリパラメータはすべて `string | string[] | undefined` のままでバリデーションなし
- yup と Zod が混在する構造になるため、スキーマの再利用ができない

### バリデーションライブラリの比較

|                      | **Zod**                              | yup              | joi              | valibot                     |
| -------------------- | ------------------------------------ | ---------------- | ---------------- | --------------------------- |
| TypeScript との統合  | ✅ `z.infer<>` で型を自動生成        | △ 手書き型が必要 | △ 手書き型が必要 | ✅                          |
| TanStack Router 統合 | ✅ `validateSearch` でネイティブ対応 | ❌               | ❌               | ❌                          |
| react-hook-form 統合 | ✅ `zodResolver`                     | ✅ `yupResolver` | △                | ✅                          |
| API レスポンス検証   | ✅ `.parse()` でランタイム検証       | ✅               | ✅               | ✅                          |
| バンドルサイズ       | △ 約 12KB                            | △ 約 19KB        | 🔴 約 24KB       | ✅ 約 8KB（tree-shakeable） |
| エコシステム         | ✅ 最大                              | ✅ 大            | △ Node.js 寄り   | △ 新しめ                    |

### Zod が決定打になった理由

**① TanStack Router の `validateSearch` とのネイティブ統合**

```ts
// Zod スキーマを渡すだけでクエリパラメータが型安全になる
const sampleModalRoute = createFileRoute('/sampleModal')({
  validateSearch: z.object({
    from: z.string(),
    serviceType: z.string(),
  }),
})
// → バリデーション失敗時は errorComponent が自動で表示される
```

**② `z.infer<>` による型の一元生成**

```ts
// スキーマを定義するだけで型が得られる（手書き interface が不要）
const userInfoResponseSchema = z.object({
  clientUserId: z.string(),
  groupId: z.string(),
  stores: z.array(z.object({ storeCode: z.string(), storeName: z.string() })),
})

type UserInfoResponse = z.infer<typeof userInfoResponseSchema>
// → スキーマと型が常に一致する（乖離が構造的に起きない）
```

**③ スキーマを複数の場所で再利用できる**

```ts
// 同じスキーマをルーティング・API・フォームで使い回せる
const storeCodeSchema = z.string().min(1)

// ルーティング
validateSearch: z.object({ storeCode: storeCodeSchema })

// API レスポンス
const storeSchema = z.object({ storeCode: storeCodeSchema, storeName: z.string() })

// フォーム（将来）
const formSchema = z.object({ storeCode: storeCodeSchema })
```

yup を継続する場合、TanStack Router との統合は自前実装が必要になり、
スキーマを再利用するには yup ↔ Zod の変換コードが必要になる。

---

## Decision

**バリデーションライブラリを Zod に統一する。ルーティング・API レスポンス・フォームの全レイヤーで同一のスキーマを使う。**

---

## Consequences

**メリット**

- TanStack Router の `validateSearch` と組み合わせることで、全ページのクエリパラメータ・パスパラメータが型安全になる
- `z.infer<>` によりスキーマから型が自動生成されるため、手書き `interface` との乖離が構造的に生まれない
- API レスポンスを `.parse()` でランタイム検証できるため、バックエンドの変更を早期に検知できる
- TypeScript `strict: true`（ADR-008）との相乗効果で、型安全性がエンドツーエンドで担保される

**デメリット・制約**

- 旧実装の yup スキーマを Zod に書き直す移行コストがかかる
- バンドルサイズが valibot より約 4KB 大きい（ただし TanStack Router との統合優位性が上回ると判断）
- Zod v4 以降は一部 API が変更されているため、ライブラリ更新時は変更内容の確認が必要
